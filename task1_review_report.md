# 0107 Code Review 报告

## 第一层：功能与边界层

1. 分类规则未真正进入运行逻辑
- 代码只把问题文本直接丢给模型，`task1_categories.md` 中的边界说明没有被用于判定。
- 影响：多意图、流程抱怨、退款进度等边界题不稳定。
- 建议：把分类节点拆成规则优先 + LLM 兜底，并把规则配置化，便于前端维护优先级和文本映射集。

2. 输出协议过于脆弱
- 代码直接信任模型返回文本。
- 影响：重复标签、尾部特殊 token、解释性文本都会污染结果。
- 证据：LLM 全量评测中，baseline v1 准确率仅 13.3%，无效输出率达 83.3%。
- 建议：输出必须做枚举校验，非法输出统一归一到兜底策略；prompt 需显式要求“只输出一个标签”。

3. 没有明确的边界题策略
- 例如“退货流程太麻烦”应优先投诉建议，“退款的事顺便看物流”应按主诉求判。
- 影响：会把客服投诉、商品咨询、退款退货混在一起。
- 建议：显式定义主诉求优先、投诉优先、退款优先等规则，并注入到 structured prompt 的 boundary rules 中。

4. 无效输入缺少兜底
- 问候、纯符号、短句、空白没有单独策略。
- 影响：模型会随机给类，或者评估不稳定。
- 建议：将这类样本统一收敛到“其他”。

## 第二层：工程与 harness 层

1. 单条异常不能拖垮整批
- 当前任何 API/解析异常都有可能中断整批。
- 建议：保留最小异常隔离、失败落盘、可重试，但不在此阶段做重并发。

2. provider 配置需要外置
- 当前接入信息不应写死在源码里。
- 建议：使用环境变量或配置文件注入。

3. 需要结果可观测性
- 建议输出 `source / reason / matched_rule / raw_output`，方便排查错例。

## 这次优先改的方向
- 先把纯分类能力做稳定，保证 30 条测试样本能跑通并提升准确率。
- 分类规则做成配置化，方便后续在前端直接维护优先级和关键词。
- Prompt 做 A/B 版本化，baseline 与 structured v2 分开评测。
- 工程化层只保留轻量兜底，不做重框架化改造。
