# 0107 · 任务1 + 任务4

## 任务1：Code Review 结论（按严重程度排序）

### 1. 明文硬编码 API Key（严重）
- 原文件：`task1_classifier.py`
- 问题：`openai.api_key` 直接写在源码里。
- 影响：密钥一旦进仓库、截图或被转发，就等于泄露；会带来费用风险、权限风险和后续轮换成本。

### 2. 单条调用失败会中断整批分类（高）
- 原文件：`task1_classifier.py`
- 问题：`batch_classify()` 循环里没有任何异常处理；任意一次 API 超时、限流、网络抖动、返回结构异常，整批任务直接失败。
- 影响：评测样本无法稳定跑完；线上批量任务可用性差，偶发报错正是这类问题的典型表现。

### 3. 模型输出未做合法性校验（高）
- 原文件：`task1_classifier.py`
- 问题：代码直接信任 `response.choices[0].message.content`，没有检查是否落在合法标签集合内。
- 影响：模型如果返回“商品问题”“物流相关”“退款/售后”这类近义词，结果会被静默写入输出，污染准确率统计，也会让下游分单失败。

### 4. Prompt 过于简陋，缺少边界规则（中）
- 原文件：`task1_classifier.py`
- 问题：只给了标签列表，没有明确“主诉求优先”“退款进度不算物流”“纯辱骂归其他、带投诉内容归投诉建议”等边界。
- 影响：对多意图样本和模糊表达不稳定，容易出现精度抖动。

### 5. 缺少基础可配置性与可测试性（中）
- 原文件：`task1_classifier.py`
- 问题：模型名写死、无环境变量、无测试、无失败记录。
- 影响：难以调参、难以回归验证，也不利于后续 prompt A/B 评估。

## 任务4：本次先完成的 2 项工程化改进

### 改进 A：安全配置外置
- 新文件：`classifier_improved.py`
- 改动：不再硬编码密钥，统一从 `OPENAI_API_KEY` 读取；模型名也改为环境变量 `CLASSIFIER_MODEL` 可配置。
- 价值：消除最直接的安全风险，也让脚本具备最基本的部署条件。

### 改进 B：批量任务韧性 + 输出校验
- 新文件：`classifier_improved.py`
- 改动：
  - 为单条分类增加最多 3 次重试和退避。
  - 对模型输出做合法标签归一化与校验。
  - 单条失败时默认降级为错误记录，而不是整批崩溃。
  - 输出中保留 `status`、`attempts`、`raw_output`、`error`，方便排查偶发问题。
- 价值：直接对应“偶发报错”和“分类不稳定”两个核心问题，是比表面 prompt 微调更高优先级的工程兜底。

## 为什么优先做这两项

- 这两项都直指可用性底线：一个解决安全事故风险，一个解决批处理稳定性和结果可控性。
- 如果不先补这层工程兜底，后面即使 prompt 调优带来一点准确率提升，线上仍会被超时、限流、脏输出拖垮。
