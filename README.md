## 第一章：Code Review

### 1.1 双层缺陷总览

这次 Review 我按两层来做：

- **第一层：功能与业务边界层**
  - 看这段代码是否真的能完成“客户诉求分类”这件事。
  - 核心问题是：分类逻辑是否成立、边界情况是否处理、输出是否稳定可用。

- **第二层：Harness / 工程层**
  - 看这段脚本能不能真正作为一个节点接入系统运行。
  - 核心问题是：稳定性、可靠性、回退机制、并发能力、可观测性是否足够。

下面按严重程度排序。

### 1.2 第一层：功能与业务边界缺陷

#### P0-1 分类规则没有真正进入运行逻辑

**问题**

原始 `classifier.py` 只有一个非常薄的 user prompt：给出 6 个标签，然后要求模型直接回复类别名称。虽然附件里有 `classification_prompt.md` 和 `categories.md`，但这些业务规则并没有真正进入运行逻辑。

**影响**

- 多意图问题不稳定
- 边界题不稳定
- 投诉类、流程抱怨、退款进度等场景容易误判

**典型样例**

- `你们这个退货流程也太麻烦了吧，我都搞不懂怎么操作`
  - 正确标签：`投诉建议`
  - 如果只看“退货”“操作”，模型很容易误判成 `退款退货`

**改进思路**

- 让分类逻辑真正使用 `categories.md` 中的业务定义与边界规则
- 明确主诉求优先
- 明确退款/物流、投诉/退货流程、问候/其他等边界

**落地实现**

- 设计了 `v2 structured` 与 `v3 implicit reasoning` 两版结构化 prompt
- 将边界条件显式化，特别是隐式退款、隐式物流等场景

---

#### P0-2 输出协议过于脆弱，结果不可直接消费

**问题**

原始代码直接信任 `response.choices[0].message.content.strip()`，没有做任何标签合法性校验，也没有控制重复标签、解释性文本、特殊 token 的污染。

**影响**

- 即使模型语义判断正确，输出格式也可能导致分单失败
- 评测准确率会被“假错误”拉低

**真实测试现象**

在 baseline 提示词下，模型会返回类似：

- `商品咨询\n\n商品咨询`
- `退款退货</s>`
- `其他\n\n其他`

这些都不是系统可以直接消费的稳定输出。

**改进思路**

- 这一块最终不是只做“字符串清洗”，而是做成一条完整的输出治理链：

1. **Prompt 层**
   - 不再让模型自由输出自然语言，而是要求输出结构化 JSON，只保留分类结果字段。

2. **后置 JSON Repair**
   - 如果模型输出不是严格 JSON，先做轻量修复：去掉代码块标记、解释性前缀、尾部特殊 token，再尽量还原为 JSON 载荷。

3. **指数退避重试**
   - 如果修复后仍无法解析或结果不合法，则按指数退避进行重试，避免偶发输出污染直接导致失败。

4. **Pydantic 校验**
   - 修复后的结果必须通过 `Pydantic` 结构校验，确认字段齐全、标签合法、不会带入非法类别。

**落地实现**

- `classifier_improved.py` 中增加：
  - `ClassificationPayload`：Pydantic 输出模型
  - `repair_json_payload()`：JSON Repair
  - `validate_payload()`：标签合法性校验
- `task1_service.py` 负责指数退避重试与最终降级兜底

---

#### P0-3 缺少主诉求判定机制

**问题**

原题要求最终只能分到一个客服组，但原始脚本没有告诉模型：一句话同时包含多个方向时，该如何取“主诉求”。

**影响**

- 多意图句子会随机落到某个局部关键词上
- 模型容易被“顺便”部分带偏

**典型样例**

- `我想问下这个退款的事顺便看看快递到没到`
  - 正确标签：`退款退货`
  - 如果没有“主诉求优先”规则，可能会误判成 `物流查询`

**改进思路**

- 在 prompt 中明确：多意图时，以主要诉求为准
- 在 few-shot 中专门加入多意图样例

**落地实现**

- `v2 structured` 和 `v3` 都明确写入 `boundary_rules`

---

#### P0-4 对隐式语义的处理不够

**问题**

原始提示词对显式语义还算有机会猜中，但对于没有明确分类词的自然化表达，理解明显不稳。

**影响**

- 遇到“只处理一件”“恢复售后流程”“门牌写错还能改吗”这类句子时，模型会在 `商品咨询 / 其他 / 物流查询 / 退款退货` 间漂移

**典型失败样例（旧版 structured prompt）**

- `售后流程开始后还能恢复吗`
  - 期望：`退款退货`
  - 误判：`商品咨询`

- `一笔里能只处理其中一件吗`
  - 期望：`退款退货`
  - 误判：`商品咨询`

- `我写的门牌错了，还来得及改吗`
  - 期望：`物流查询`
  - 误判：`其他`

**改进思路**

- 将“潜在语义还原”正式写进 prompt 规范，而不是只靠示例暗示
- 新增 `boundary_conditions` 章节，要求模型先还原业务场景，再判断类别

**落地实现**

- 新增 `prompts/classification_prompt_v3_implicit.md`
- 显式写入：
  - `处理 / 售后 / 恢复 / 其中一件` 往往隐含退款退货语境
  - `门牌 / 门号 / 楼栋 / 填错了 / 改地址` 往往隐含物流语境

---

#### P1-5 无效输入没有兜底策略

**问题**

原始脚本没有区分问候、纯标点、超短模糊句等无效输入。

**影响**

- 模型可能随机猜类
- 会给客服分单系统带来噪声

**典型样例**

- `？？？`
- `你好`

**改进思路**

- 在 prompt 中明确：纯问候、纯标点、无法判断归 `其他`
- 在规则层加兜底

## 第二章：Prompt 改进与评测

### 2.1 Prompt 总体设计思路

原始 prompt 只有一句话：给标签列表，让模型直接选一个。问题在于：

- 输出协议不稳
- 边界规则不清
- 多意图无机制
- 隐式语义几乎无处理

所以本次 Prompt 改进的思路是：

1. 从 `v1 baseline` 建一个最小基线
2. 用 `v2 structured（还原版）` 解决输出与基础边界问题
3. 用 `v3 implicit reasoning` 解决隐式语境和情境还原问题

---

### 2.2 Prompt 结构

三版 Prompt 的核心结构分别如下：

#### v1 baseline

- 只有单条 user prompt
- 没有 system prompt
- 没有结构化分层
- 没有 few-shot
- 没有显式边界条件

#### v2 structured（还原版）

- system / user 分离
- XML 分层
- 分类定义 `categories`
- 基础边界规则 `boundary_rules`
- 少量基础 few-shot

#### v3 implicit reasoning

- 保留 v2 的结构化设计
- 新增 `boundary_conditions` 章节
- 显式说明隐式语境的业务还原
- 给出潜在含义推理原则
- 用隐式 few-shot 对应强化

---

### 2.3 Prompt 版本演进表

| 版本 | 设计特点 | 解决的问题 | 已知短板 | 关键结果 |
|---|---|---|---|---|
| v1 baseline | 单条 user prompt，只有标签列表和“只回复类别名称” | 最小可运行基线 | 输出协议脆弱；边界规则缺失；多意图和隐式语义几乎无约束 | 30 条金标：13.3%，无效输出率 83.3% |
| v2 structured（还原版） | system/user 分离；XML 分层；分类定义；基础边界规则；少量 few-shot | 解决输出失控、基础边界不清问题 | 对隐式投诉、隐式退款、隐式物流理解仍不稳定 | 20 条隐式边界集：95.0% |
| v3 implicit reasoning | 在 v2 基础上新增 `boundary_conditions`；显式写潜在语义推理；补充隐式 few-shot | 解决“没有显式分类词时如何还原语境”的问题 | prompt 更长、token 成本更高 | 30 条金标 llm-only：100.0%；120 条增强集 llm-only：100.0% |

---

### 2.4 各版本测试情况与改动理由

#### v1 baseline

**测试集**

- 原始金标集：30 条

**结果**

- 准确率：`13.3%`
- 无效输出率：`83.3%`

**暴露的问题**

- 模型经常输出重复标签、解释文本、非标准格式
- 分类结果不稳定
- 边界题靠猜

**结论**

- v1 证明：只靠薄 prompt 不足以完成真实客服分单

---

#### v2 structured（还原版）

**改动理由（从上一版测试暴露的问题出发）**

- v1 的主要问题不是单纯“语义不懂”，而是“输出协议失控”和“边界规则缺失”
- 所以 v2 首先解决：
  - 输出只允许一个标签
  - system / user 分离
  - 加入 categories 与基础 boundary rules

**改动理由（从代码和情境分析出发）**

- 客服 FAQ 是一个非常标准的分类问题，天然适合结构化 prompt
- 原始代码和附件文档脱节，因此 v2 要先把文档规则真正用起来

**测试结果**

- 30 条金标：`100.0%`
- 20 条新增隐式边界集：`95.0%`

**失败样例**

- `整个处理过程都让我有点崩溃`
  - 正确标签：`投诉建议`
  - v2 预测：`其他`
  - 原因：没有“投诉/建议/质量/流程麻烦”这类显式词时，旧版容易保守兜底

**结论**

- v2 已经把基础 structured 分类能力做起来了
- 但对隐式投诉、隐式退款、隐式物流的还原仍不够显式

---

#### v3 implicit reasoning

**改动理由（从上一版测试暴露的问题出发）**

- v2 在新增 20 条边界集上仍有误判，说明 few-shot 还不够，必须把“隐式语境推理”写成规范

**改动理由（从代码和情境分析出发）**

- 用户真实表达往往不会把“退款、物流、咨询”说得很标准
- 更常见的是：
  - `一笔里能只处理其中一件吗`
  - `售后流程开始后还能恢复吗`
  - `我写的门牌错了，还来得及改吗`
- 这些句子需要模型先还原业务情境，再判断类别

**新增内容**

- `boundary_conditions` 章节
- “潜在语义推理原则”
- 隐式退款 / 隐式物流的正确推理示例

**测试结果（规则关闭）**

- 30 条金标 llm-only：`100.0%`
- 20 条隐式边界集 llm-only：`100.0%`
- 120 条增强压力集 llm-only：`100.0%`

**压力测试指标（120 条，llm-only，并发 12）**

- 样本数：`120`
- 并发度：`12`
- 总准确率：`100.0%`
- 无效输出率：`0.0%`
- LLM 命中率：`100.0%`
- 降级回退率：`0.0%`
- 总耗时：`46.7s`
- 吞吐：`2.57 req/s`
- 平均响应时长：`4420.74 ms`
- P95：`5831.10 ms`
- P99：`6680.80 ms`
- Total Tokens：`164091`
- 平均单请求 Tokens：`1367.42`

**指标说明**

- `accuracy`：总体分类准确率
- `invalid_output_rate`：非法标签或脏输出占比
- `llm_hit_rate`：真正由 LLM 完成分类的比例
- `degraded_fallback_rate`：服务退化后兜底的比例
- `throughput_rps`：单位时间处理请求数
- `P95/P99`：高位时延，反映高并发下尾延迟表现
- `Total Tokens / avg_total_tokens_per_request`：模型成本与单请求消耗

**结论**

- v3 是本次最完整、最适合作为正式沉淀版本的 Prompt
- 它不仅提升了效果，也把“为什么能分对”写成了规范

## 第三章：工程化改进

题目要求至少 2 项工程化改进；本次实际完成了 4 类，而且都围绕格式校验、稳定性、并发性与鲁棒性：

### 3.1 输出结果的结构化治理链

- 为解决“重复标签、解释性文本、特殊 token 污染、非法类别”等问题，分类结果治理被拆成四层：
  1. Prompt 强约束 JSON
  2. JSON Repair
  3. 指数退避重试
  4. Pydantic 校验

**价值**

- 输出不再依赖模型“自觉听话”，而是具备程序化校验与恢复能力
- 能明显降低协议层脏输出对分类结果的污染

---

### 3.2 规则配置化与 BAD CASE 可扩展

- 规则文件外置为 `task1_rules.json`
- 支持维护：
  - 6 类优先级
  - 每类关键词映射
  - 特定 BAD CASE 的临界词补充

**价值**

- 不需要改核心代码即可增补新规则
- 后续可以持续将线上错例和 BAD CASE 回灌到规则层

---

### 3.3 服务级稳定性、回退与鲁棒性

在 `task1_service.py` 中补了节点运行层的完整外壳：

- 重试
- 服务降级
- 熔断器
- 批量并发
- 可观测字段输出

**价值**

- 节点不再只是一个脚本，而是可接入系统的分类工具

---

### 3.4 并发批处理与压力测试验证

- 使用 `ThreadPoolExecutor` 支持受控并发批量分类
- 对 `llm_only` 模式做了 100 条和 120 条压力测试
- 指标覆盖：
  - 总准确率
  - 无效输出率
  - LLM 命中率
  - 降级回退率
  - 平均时延 / P95 / P99
  - token 消耗

**价值**

- 证明节点不仅能“分对”，还能在并发条件下稳定运行
- 给后续接入真实系统提供性能基线

### 4.3 本次 AI 协作的关键思路线索

这部分不是完整对话摘抄，而是保留几个关键决策节点：

1. **先纠偏任务重心**
   - 一开始不是急着补 try/except 或安全性，而是先确认：这题本质上考的是“客户诉求分类功能能不能真的用”。

2. **先区分两层问题**
   - 第一层是分类功能本身：边界、主诉求、隐式语义。
   - 第二层才是 harness：稳定性、回退、并发、可观测性。

3. **先用真实 API 跑出 baseline**
   - 不是空谈 Prompt 优化，而是先验证 baseline 到底差在哪里。
   - 很快确认：旧版的核心问题除了语义，还有输出协议失控。

4. **Prompt 不是只加 few-shot，而是版本化演进**
   - 从 v1 到 v2，再到 v3，不是简单改词，而是把边界规则和潜在语义推理逐步沉淀成规范。

5. **规则和 LLM 两条线分开验证**
   - 规则版证明节点落地能力
   - LLM-only 版证明模型自身的分类能力与降级能力

6. **最后再把它补成一个完整工具节点**
   - 配置化规则
   - A/B 评测
   - 服务层保护
   - 压测与回归验证
