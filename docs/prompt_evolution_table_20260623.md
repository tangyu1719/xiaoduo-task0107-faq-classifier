# Prompt 演进表（v1 -> v2 还原版 -> v3）

| 版本 | 设计特点 | 解决的问题 | 已知短板 | 关键结果 |
|---|---|---|---|---|
| v1 baseline | 单条 user prompt，只有标签列表和“只回复类别名称” | 最小可运行基线 | 输出协议脆弱；边界规则缺失；多意图和隐式语义几乎无约束 | 30 条金标：13.3%，无效输出率 83.3% |
| v2 structured（还原版） | system/user 分离；XML 分层；分类定义；基础边界规则；少量 few-shot | 解决输出失控、基础边界不清问题 | 对隐式投诉、隐式退款、隐式物流理解仍不稳定 | 20 条隐式边界集：95.0% |
| v3 implicit reasoning | 在 v2 基础上新增 `boundary_conditions` 章节；显式写潜在语义推理；补充隐式 few-shot | 解决“没有显式分类词时如何还原语境”的问题 | prompt 更长、token 成本更高 | 30 条金标 llm-only：100.0%；120 条压力集 llm-only：100.0% |

## 版本演进说明

### v1 -> v2
- 核心升级：从“自然语言提示”升级为“结构化规范”。
- 重点修复：
  - system / user 不分离
  - 输出不受控
  - 标签边界不明确

### v2 -> v3
- 核心升级：从“示例告诉模型怎么做”升级为“显式规定隐式语境推理规则”。
- 重点修复：
  - “处理/售后/恢复”等弱指向词的语义还原
  - “门牌/楼栋/填错了”等隐式物流语义识别
  - 对无显式分类词句子的业务场景补全

## 结论
- `v1` 证明：只靠薄 prompt，输出协议和分类边界都会出问题。
- `v2` 证明：结构化 prompt 已经能大幅提升基础能力。
- `v3` 证明：当任务开始考验“隐式语义 / 边界条件 / 情境还原”时，必须把推理规范写进 prompt，而不只是增加 few-shot。
